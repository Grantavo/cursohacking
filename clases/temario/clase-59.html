Operando con proyectos de Github [4-5]

Duración: 2:02 · Sección 2: Temario

Los tags y releases son mecanismos para marcar puntos específicos en la historia del repositorio, normalmente para versiones estables del software.

Tags en Git

Un tag es una referencia a un commit concreto. Se usa para marcar versiones (v1.0, v2.3.1, etc.):

# Crear un tag ligero (solo un nombre)
git tag v1.0.0

# Crear un tag anotado (con mensaje, autor, fecha)
git tag -a v1.0.0 -m "Versión estable 1.0.0"

# Crear un tag en un commit específico
git tag -a v1.0.0 <commit-hash> -m "Versión 1.0.0"

# Listar todos los tags
git tag
git tag -l "v2.*"  # Filtrar por patrón

# Mostrar información de un tag
git show v1.0.0

Subir tags a GitHub

Los tags no se suben automáticamente con git push. Hay que hacerlo explícitamente:

# Subir un tag específico
git push origin v1.0.0

# Subir todos los tags locales que no están en el remoto
git push origin --tags

git describe genera un nombre descriptivo del commit actual basado en el tag más cercano:

git describe
# v1.0.0-5-g3a4b5c6
# Significa: sobre v1.0.0, 5 commits después, hash g3a4b5c6

Releases en GitHub

En la interfaz web de GitHub, los tags se convierten en Releases donde puedes adjuntar binarios, notas de versión y changelogs. Se accede desde la pestaña Releases del repositorio.

Buenas prácticas en repositorios

Un repositorio bien mantenido incluye ciertos archivos fundamentales:

ArchivoPropósito
README.mdDescripción del proyecto, cómo instalarlo, usarlo y ejemplos
LICENSELicencia bajo la que se distribuye el software (MIT, GPL, Apache, etc.)
CONTRIBUTING.mdGuía para contribuir: estándares de código, cómo hacer PRs, tests
CHANGELOG.mdRegistro de cambios por versión
.gitignoreArchivos que Git debe ignorar (binarios, dependencias, .env)
📖 El archivo README.md es lo primero que ve cualquiera que visita tu repositorio. Incluye instrucciones claras de instalación, ejemplos de uso, y enlaces a documentación adicional.
